## Hardened Docker Runtime for Cards & Payments

На данном этапе была реализована защищённая среда выполнения (secure runtime) для сервисов **Cards** и **Payments**.

Цель шага — перевести сервисы из обычного запуска контейнера в режим повышенной безопасности, соответствующий принципам DevSecOps.

### Реализованные меры защиты

- Контейнеры запускаются не от root (`--user UID:GID`)
- UID контейнера согласован с сервисным пользователем на хосте
- Файловая система контейнера работает в режиме `--read-only`
- Конфигурационные файлы монтируются только для чтения (`:ro`)
- Применён `--security-opt no-new-privileges`
- Удалены все Linux capabilities (`--cap-drop ALL`)
- Добавлены ограничения по ресурсам (CPU, memory, pids)
- Используется изолированная Docker-сеть

### Проведена валидация

- Контейнеры находятся в состоянии `Running`
- Логи приложения продолжают генерироваться (heartbeat)
- Конфигурационные файлы читаются внутри контейнера
- Попытка изменения конфигурации приводит к `Permission denied`
- Процесс внутри контейнера работает под ожидаемым UID

### Результат

Сервисы переведены в hardened runtime режим.

Компрометация контейнера не позволяет:
- повысить привилегии
- изменить конфигурацию
- записать вредоносные изменения
- выйти за пределы своего UID
- автоматически получить доступ к другому сервису

Таким образом реализовано снижение blast radius и практическое применение принципа Least Privilege.
